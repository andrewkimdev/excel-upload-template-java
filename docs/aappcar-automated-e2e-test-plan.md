# AAPPCAR 자동 E2E 사전검증 계획

연계 문서: [AAPPCAR 수동 E2E 업로드 테스트 및 산출물 계획](./aappcar-manual-e2e-test-plan.md)

## 1. 문서 목적

이 문서는 [`aappcar-manual-e2e-test-plan.md`](./aappcar-manual-e2e-test-plan.md)를 사람이 수행하기 전에, 반복 가능하고 자동화 가능한 검증 항목을 먼저 실행해 수동 테스트 시간을 줄이기 위한 자동 E2E 사전검증 계획을 정의한다.

본 문서는 아직 구현되지 않은 자동 테스트 스위트의 설계 문서이며, 코드 구현 범위와 기대 산출물을 명확히 남기는 것을 목표로 한다.

## 2. 자동화 목표

- 수동 시나리오와 1:1로 대응되는 자동 검증 기준 수립
- 사람이 직접 만들 필요가 없는 테스트 입력 파일 자동 생성
- 오류 리포트 다운로드 및 기본 품질 자동 확인
- 수동 테스터가 바로 재사용할 수 있는 입력 파일/결과 파일/요약본 생성
- 수동 테스트 전에 서버 계약과 주요 UI 렌더링 흐름의 위험요소를 조기 발견

## 3. 자동화 원칙

- 브라우저 자동화 도구는 도입하지 않는다.
- 현재 저장소의 표준 테스트 방식인 `MockMvc + Spring Boot 통합 테스트 + Apache POI`를 사용한다.
- REST 업로드 API를 주 검증 경로로 사용하되, 수동 문서의 UI 중심 목적을 반영해 서버 렌더링 폼 흐름도 필요한 범위까지 함께 검증한다.
- 문서와 런타임 구현이 충돌하면 실제 코드 동작을 기준으로 판단하되, 충돌 사실은 자동 결과 요약에 남긴다.
- 자동 생성 산출물은 저장소에 커밋하지 않고 테스트 실행 산출물로만 남긴다.

## 4. 계획 범위

### 4.1 주 검증 대상

- 업로드 화면 GET: `/upload/aappcar`
- 업로드 API: `/api/excel/upload/aappcar`
- 업로드 화면 POST 결과 렌더링: `/upload/aappcar`
- 오류 파일 다운로드 API: `/api/excel/download/{fileId}`

### 4.2 자동화가 직접 줄여야 할 수동 작업

- 시나리오별 Excel 입력 파일 제작
- 기본 metadata 반복 입력 검증
- 실패 시 오류 리포트 다운로드 확인
- `_ERRORS` 컬럼 존재 여부 확인
- 동일 시나리오의 반복 재실행 및 결과 정리

## 5. 구현 접근 계획

### 5.1 테스트 스위트 형태

- `src/test/java/com/foo/excel/integration/` 아래에 수동 계획 대응용 전용 통합 테스트 스위트를 추가한다.
- 테스트는 실제 Spring 컨텍스트에서 실행한다.
- 각 시나리오는 독립 실행 가능해야 하며, 필요한 선행 데이터는 테스트 내부에서 직접 준비한다.

### 5.2 입력 파일 생성 방식

- 각 TC에 대응하는 파일을 테스트 런타임에 생성한다.
- 파일명은 수동 문서의 규칙을 그대로 따른다.
  - `TC-XX_<slug>_input.xlsx`
  - `TC-XX_<slug>_input.xls`
- workbook 생성은 공통 helper로 관리해 헤더 구조, 병합, footer marker, 기본 데이터 구성을 재사용한다.
- 생성된 입력 파일은 사람이 수동 테스트에 재사용할 수 있도록 별도 산출물 디렉터리에 저장한다.

### 5.3 결과 산출물 생성 방식

자동 테스트 실행 시 아래 구조를 생성한다.

```text
build/e2e-aappcar/
  01_test-files/
  02_execution-results/
  03_error-reports/
  04_summary/
```

- `01_test-files/`: 각 TC 입력 파일 저장
- `02_execution-results/`: 각 TC 결과 Markdown 저장
- `03_error-reports/`: 시스템이 생성한 오류 리포트 저장
- `04_summary/`: 전체 결과 요약 저장

## 6. 시나리오 대응 계획

자동 스위트는 수동 문서의 아래 시나리오를 동일한 ID로 대응한다.

| ID | 자동 검증 계획 |
|---|---|
| `TC-01` | 정상 업로드 성공, DB 반영, 오류 리포트 미생성 확인 |
| `TC-02` | `.xls` 업로드 차단 확인 |
| `TC-03` | 확장자 위장 비정상 파일 차단 확인 |
| `TC-04` | 필수 metadata 누락 검증 확인 |
| `TC-05` | 승인 장비 메타데이터 중복 사전 차단 확인 |
| `TC-06` | 헤더 불일치 실패 확인 |
| `TC-07` | 필수 컬럼 누락 실패 확인 |
| `TC-08` | 행 필수값 누락 시 오류 리포트 생성 확인 |
| `TC-09` | 데이터 형식 오류 시 오류 리포트 생성 확인 |
| `TC-10` | 파일 내 중복 데이터 검증 실패 및 오류 리포트 생성 확인 |
| `TC-11` | 오류 리포트 다운로드 성공 및 `_ERRORS` 존재 확인 |
| `TC-12` | 최대 행 수 초과 차단 확인 |

## 7. 자동 검증 포인트

### 7.1 공통 검증

- 기대 HTTP 상태 코드
- 기대 한국어 메시지
- 성공/실패 여부
- 오류 리포트 생성 여부
- 다운로드 URL 또는 `errorFileId` 존재 여부
- 필요한 경우 DB 반영 여부

### 7.2 실패 시점 검증

수동 테스트 시간을 줄이기 위해, 단순 실패 여부뿐 아니라 기대한 시점에서 실패하는지도 확인한다.

- 요청/metadata 검증 단계 실패
- 파일 보안 검증 단계 실패
- import precheck 단계 실패
- 헤더/컬럼 해석 단계 실패
- 행 검증 단계 실패
- 최대 행 수 제한 단계 실패

### 7.3 UI 관련 최소 검증

브라우저 자동화는 하지 않지만, 수동 계획의 UI 목적을 반영해 아래는 자동 검증한다.

- `/upload/aappcar` 진입 가능 여부
- form POST 성공 시 결과 화면 렌더링
- form POST 실패 시 결과 메시지 렌더링
- 오류 리포트가 있는 실패 결과에서 다운로드 링크 렌더링

## 8. 자동 결과 기록 형식

각 자동 실행 결과 파일은 수동 문서의 기록 형식을 최대한 재사용한다.

```text
시나리오 ID:
시나리오명:
실행일시:
실행 환경:
입력 파일명:
입력 metadata 값:
실제 HTTP 상태:
실제 화면 또는 API 메시지:
오류 리포트 생성 여부:
시스템 생성 결과 파일명:
DB/데이터 확인 결과:
기대 대비 판정: PASS / FAIL
특이사항:
```

## 9. 총괄 요약본 계획

`04_summary/manual-e2e-test-summary.md`에 아래 항목을 포함한다.

- 전체 시나리오 수
- 수행 완료 수
- PASS 수
- FAIL 수
- 미수행 수
- 자동으로 확인된 주요 위험요소
- 수동 테스터가 추가로 볼 만한 항목
- 생성 산출물 목록

## 10. 구현 시 주의사항

- 자동화는 수동 테스트를 대체하는 것이 아니라 사전 위험 제거 수단으로 구현한다.
- 사람이 실제로 열어볼 수 있는 파일 기반 산출물을 남겨야 한다.
- 시나리오 ID와 산출물 파일명이 반드시 1:1로 연결되어야 한다.
- 구현 전까지는 본 문서를 기준 계획으로 유지하고, 실제 코드 변경은 별도 작업으로 진행한다.
