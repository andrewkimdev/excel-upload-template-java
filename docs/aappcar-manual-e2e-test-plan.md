# AAPPCAR 수동 E2E 업로드 테스트 및 산출물 계획

연계 문서: [AAPPCAR 자동 E2E 사전검증 계획](./aappcar-automated-e2e-test-plan.md)

## 1. 문서 목적

이 문서는 `aappcar` 업로드 기능에 대해 수동 E2E 테스트를 수행하기 위한 실행 계획과 제출 산출물 구성을 정의한다.

요청 범위는 아래를 모두 포함한다.

- 정상 업로드
- 비정상 업로드
- 파일 형식 오류
- 필수 컬럼 누락
- 데이터 형식 오류
- 기타 비정상 케이스
- 테스트 이후 전달할 산출물 정리

본 문서는 현재 저장소의 실제 런타임 계약을 기준으로 작성되었으며, 다른 프로젝트로 이식해 사용할 때는 환경값과 사전조건만 현장에 맞게 치환한다.

## 2. 테스트 대상 및 기준

### 2.1 대상 기능

- 업로드 화면: `/upload/aappcar`
- 업로드 API: `/api/excel/upload/aappcar`
- 오류 파일 다운로드 API: `/api/excel/download/{fileId}`

### 2.2 현재 동작 기준

- 업로드 파일은 `.xlsx`만 허용
- `.xls` 업로드는 실패해야 함
- 파일명/파일 내용 보안 검증 수행
- 메타데이터는 업로드 시 필수
- 헤더 불일치 또는 필수 컬럼 누락 시 업로드 실패
- 행 단위 검증 실패 시 오류 리포트 `.xlsx` 생성
- 오류 리포트에는 `_ERRORS` 컬럼이 포함되어야 함
- 승인된 장비 메타데이터 중복 시 파싱 전 사전 차단

### 2.3 시연 채널

- 기본 시연 채널은 웹 UI
- 필요 시 결과 확인은 서버 응답, 다운로드 파일, DB 또는 운영 로그로 보조 확인

## 3. 테스트 환경 정의

문서 작성 및 실제 수행 시 아래 항목을 먼저 고정한다.

| 항목 | 내용 |
|---|---|
| 실행 환경 | 로컬 / 개발 / 검증 중 택1 |
| 서버 기동 방식 | `./gradlew bootRun` 또는 이식 대상 프로젝트 실행 명령 |
| 대상 URL | `/upload/aappcar` |
| 다운로드 URL | `/api/excel/download/{fileId}` |
| 브라우저 | 실제 시연에 사용할 브라우저 1종 이상 |
| 계정/권한 | 필요 시 명시 |
| DB 확인 방식 | 직접 조회 가능 시 조회, 불가 시 화면/응답/로그로 대체 |
| 결과 파일 보관 위치 | 테스트 수행 전 미리 생성한 산출물 폴더 |

### 3.1 기본 입력 metadata 값

아래 값은 별도 언급이 없는 한 공통 기본값으로 사용한다.

| 필드 | 값 |
|---|---|
| `comeYear` | `2026` |
| `comeOrder` | `001` |
| `uploadSeq` | `1` |
| `equipCode` | `EQ-01` |
| `equipMean` | `설비A` |
| `hsno` | `8481802000` |
| `spec` | `규격A` |
| `taxRate` | `8.50` |

## 4. 제출 산출물 구조

최종 제출 산출물은 아래 4개 폴더 구조를 기준으로 정리한다.

```text
01_test-files/
02_execution-results/
03_error-reports/
04_summary/
```

### 4.1 파일 구분

- 테스트용 입력 파일: 업로드에 사용할 수동 제작 Excel 파일
- 실행 기록 파일: 시나리오 수행 결과를 기록한 문서
- 시스템 생성 결과 파일: 시스템이 생성한 오류 리포트 파일
- 총괄 요약본: 전체 수행 결과 요약 문서

### 4.2 파일 네이밍 규칙

| 구분 | 파일명 규칙 |
|---|---|
| 입력 파일 | `TC-XX_<slug>_input.xlsx` |
| 확장자 오류 입력 파일 | `TC-XX_<slug>_input.xls` |
| 실행 기록 파일 | `TC-XX_<slug>_result.md` |
| 시스템 생성 결과 파일 | `TC-XX_<slug>_error-report.xlsx` |
| 화면 캡처 선택 파일 | `TC-XX_<slug>_screen.png` |
| 총괄 요약본 | `manual-e2e-test-summary.md` |

## 5. 공통 시연 절차

모든 시나리오는 아래 절차를 기준으로 수행한다.

1. 대상 서버를 기동한다.
2. `/upload/aappcar` 접속 후 화면 진입을 확인한다.
3. 시나리오별 사전조건을 맞춘다.
4. 시나리오에서 지정한 metadata 값을 입력한다.
5. 시나리오에서 지정한 테스트 파일을 업로드한다.
6. 화면 메시지와 처리 결과를 기록한다.
7. 실패 시 오류 파일 다운로드 가능 여부를 확인한다.
8. 오류 파일이 생성되면 `03_error-reports/` 아래에 지정한 이름으로 저장한다.
9. 사용한 입력 파일을 `01_test-files/`에 정리한다.
10. 시나리오 실행 기록을 `02_execution-results/`에 작성한다.
11. 필요 시 DB 반영 여부 또는 로그/관리 화면 기준 증빙을 남긴다.

현장 환경에서 추가 예외 케이스가 발견되면 동일 형식으로 `TC-XX`를 추가하고, 입력 파일/실행 기록/오류 파일/총괄 요약본에 함께 반영한다.

## 6. 입력 파일 설계 기준

각 입력 파일은 아래 설계 정보를 함께 기록한다.

- 기준 원본 파일명
- 데이터 행 수
- 수정 셀 주소 또는 제거 대상 컬럼
- 수정 전 값
- 수정 후 값
- 헤더 병합 유지 여부
- 푸터 `※` 유지 여부
- 기대 실패 유형 또는 기대 오류 열

예시:

- `TC-08_missing-required-cell_input.xlsx`
  - 수정 셀: `C7`
  - 수정 내용: `물품명` 공란 처리
- `TC-09_invalid-format_input.xlsx`
  - 수정 셀: `F7`
  - 수정 내용: `bad-hs-code`
- `TC-10_duplicate-composite-key_input.xlsx`
  - 수정 셀: `C7/D7/F7`, `C8/D8/F8`
  - 수정 내용: 두 행의 조합값 동일하게 설정
- `TC-06_wrong-header_input.xlsx`
  - 수정 셀: 헤더 행 `C4`
  - 수정 내용: `물품명`을 `WRONG_C`로 변경
- `TC-07_missing-required-column_input.xlsx`
  - 수정 대상: 필수 컬럼 헤더 또는 필수 컬럼 구조 제거

## 7. 테스트 시나리오

### 7.1 필수 시나리오

| ID | 시나리오명 | 목적 | 사전조건 | 테스트용 파일 | 기대 결과 |
|---|---|---|---|---|---|
| `TC-01` | 정상 업로드 신규 생성 | 유효한 `.xlsx`와 필수 metadata로 정상 업로드 확인 | 동일 키 데이터 미존재 | `TC-01_valid-upload_input.xlsx` | 성공 메시지, 오류 파일 없음, 신규 반영 |
| `TC-02` | 파일 형식 오류 `.xls` | `.xls` 차단 확인 | 없음 | `TC-02_invalid-extension_input.xls` | `.xlsx` 전용 오류 |
| `TC-03` | 파일 내용 위장 | 확장자는 `.xlsx`지만 실제 Excel이 아닌 파일 차단 확인 | 없음 | `TC-03_fake-xlsx_input.xlsx` | 파일 보안 검증 실패 |
| `TC-04` | 필수 metadata 누락 | 필수 메타데이터 검증 확인 | 없음 | `TC-04_valid-file_missing-metadata_input.xlsx` | 입력 차단 또는 업로드 실패 |
| `TC-05` | 승인 장비 메타데이터 중복 | 승인 장비 사전 차단 확인 | 동일 조합 승인 장비 선행 생성 | `TC-05_valid-file_duplicate-metadata_input.xlsx` | 즉시 실패, 오류 파일 없음, DB 신규 반영 없음 |
| `TC-06` | 헤더 불일치 | 다른 템플릿 또는 헤더명 훼손 파일 차단 확인 | 없음 | `TC-06_wrong-header_input.xlsx` | 헤더 불일치 실패 |
| `TC-07` | 필수 컬럼 누락 | 요청문 예시의 필수 컬럼 누락 직접 검증 | 없음 | `TC-07_missing-required-column_input.xlsx` | 파싱 또는 컬럼 해석 실패 |
| `TC-08` | 행 필수값 누락 | 필수 셀 공란 시 오류 리포트 생성 확인 | 없음 | `TC-08_missing-required-cell_input.xlsx` | 실패, 오류 리포트 생성, `_ERRORS` 포함 |
| `TC-09` | 데이터 형식 오류 | 형식 오류 검증 확인 | 없음 | `TC-09_invalid-format_input.xlsx` | 실패, 오류 리포트 생성 |
| `TC-10` | 파일 내 중복 데이터 | 기타 비정상 케이스로서 중복 데이터 검증 확인 | 없음 | `TC-10_duplicate-composite-key_input.xlsx` | 실패, 오류 리포트 생성 |
| `TC-11` | 오류 리포트 다운로드 검증 | 다운로드 및 결과 파일 기본 품질 확인 | 오류 리포트 생성 시나리오 1건 선행 | `TC-11_reuse-error-report-source_input.xlsx` 또는 선행 파일 재사용 | 다운로드 성공, `.xlsx`, `_ERRORS` 존재 |

### 7.2 권장 시나리오

| ID | 시나리오명 | 목적 | 사전조건 | 테스트용 파일 | 기대 결과 |
|---|---|---|---|---|---|
| `TC-12` | 최대 행 수 초과 | 기타 비정상 케이스로서 행 수 제한 차단 확인 | 환경별 최대 행 수 확인 필요 | `TC-12_too-many-rows_input.xlsx` | 최대 행 수 초과 메시지 |

## 8. 시나리오별 상세 작성 템플릿

각 시나리오는 아래 형식으로 문서에 상세 작성한다.

### TC-XX 시나리오명

| 항목 | 내용 |
|---|---|
| 목적 | 무엇을 검증하는지 |
| 우선순위 | 필수 / 권장 |
| 사전조건 | DB 상태, 선행 데이터, 환경 조건 |
| 입력 파일 | 파일명 |
| metadata 입력값 | 각 필드별 실제 입력값 |
| 파일 수정 내용 | 어떤 셀/컬럼을 어떻게 변경했는지 |
| 수행 절차 | 실제 클릭/입력/업로드 절차 |
| 기대 화면 결과 | 성공/실패 메시지 |
| 기대 서버/다운로드 결과 | 오류 파일 생성 여부, 다운로드 가능 여부 |
| 기대 데이터 결과 | 신규 생성/반영 없음/검증 실패 등 |
| 실행 기록 파일 | `TC-XX_<slug>_result.md` |
| 시스템 생성 결과 파일 | 필요 시 `TC-XX_<slug>_error-report.xlsx` |

## 10. 총괄 요약본 템플릿

`04_summary/manual-e2e-test-summary.md`에는 아래 항목을 포함한다.

- 전체 시나리오 수
- 수행 완료 수
- PASS 수
- FAIL 수
- 미수행 수
- 주요 실패 원인
- 재시험 필요 항목
- 첨부 산출물 목록

## 11. 시나리오별 검증 포인트

### 11.1 성공 시나리오 검증 포인트

- 업로드 완료 메시지 표시
- 오류 리포트 미생성
- 필요 시 DB 또는 로그에서 데이터 반영 확인

### 11.2 실패 시나리오 검증 포인트

- 기대한 실패 메시지 표시
- 기대한 시점에서 실패하는지 확인
- 오류 리포트 생성 대상 시나리오는 다운로드 링크 또는 다운로드 가능 여부 확인

### 11.3 오류 리포트 검증 포인트

- 다운로드 가능 여부
- 파일 확장자 `.xlsx`
- `_ERRORS` 컬럼 존재

## 12. 수행 시 유의사항

- UI 중심 시연이므로 브라우저 입력 검증과 서버 검증을 함께 기록한다.
- DB 직접 조회가 어려운 환경이면 화면, 응답, 운영 로그를 근거로 대체한다.
- 다른 프로젝트로 이식하면서 metadata 기본값이나 사전 데이터 생성 방식이 달라질 수 있으므로 실행 전 현장 값으로 조정한다.
- 테스트 파일과 결과 파일은 반드시 시나리오 ID 기준으로 정리해 제출 문서와 1:1 매칭되도록 한다.
